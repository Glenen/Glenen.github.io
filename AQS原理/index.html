AQS原理

发表于 2017-04-23 | 分类于 JDK | |

摘要

AQS是AbstractQueuedSynchronizer的简称，即抽象队列同步器。该类是Java并发包中大部分同步器的基础构件，如ReentrantLock、ReentrantReadWriteLock等，利用AQS可以很方便的创建锁和同步器。

AQS同步框架的出现，使得拥有更多高级特性的锁出现，如reentrantLock相比于synchronized所拥有的等待可中断、实现公平锁(默认非公平)、锁绑定多个条件高级特性，都是基于AQS而实现。虽然相比于synchronized，reentrantLock拥有了更多的特性，当我们需要用到这些特性的时候，reentrantLock确实是个很好的选择，但是在两者都能完成任务的情况下，基于性能的考虑，我们还是应该优先选择synchronized(深入理解java虚拟机著作中所建议)。

前面一直也时不时看别人关于AQS的博客，但是始终感觉不合自己的口味，估计还是喜欢自己去总结和研究。正好最近也不大忙，就抽出时间自己也来写一篇关于AQS的博客，加深自己的理解，也当自己的随记吧。文中仅代表自己观点，仅供参考。

底层数据结构

通过我们对AQS的内部节点Node的观察，AQS底层是基于一个双向的链表数据结构实现。由该双向链表来实现了一个FIFO队列，FIFO队列本身也体现着一种公平性(先进去的先出来)。同时由于链表相较于数组的插入和删除高效性(不需移动元素，只改变指针的指向)，也很好的契合了队列的删除节点的需求。如下图所示：

AQS实现原理

由于AQS只是提供了同步的基础构件，就如何实现锁的获取和释放细节都需要继承的子类自己去实现。其实就该灵活的扩展设计思想也是非常值得学习和借鉴。为了更好的讲述AQS的原理，我就选择了ReentrantLock类中非公平锁NonfairSync结合讲解和理解。
